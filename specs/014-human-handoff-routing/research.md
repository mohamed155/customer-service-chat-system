# Research: Human Handoff & Routing

**Feature**: 014-human-handoff-routing | **Date**: 2026-07-14

All Technical Context unknowns resolved. Each decision below records what was chosen, why, and what was rejected.

## R1. Real-time transport: SSE over fetch (not WebSocket, not polling)

- **Decision**: One tenant-scoped Server-Sent Events endpoint, `GET /tenant/events`, implemented with `axum::response::sse` and consumed by a fetch-based streaming client in Angular (not the native `EventSource`).
- **Rationale**: The clarified requirements (FR-025, SC-008, FR-017a) need server→client push only — assignment notifications, queue/availability freshness, and a presence signal. SSE covers all three with plain HTTP: it passes through the existing tower middleware stack (auth cookie, CSRF origin check, tenant-context, request-id tracing) unchanged, auto-reconnects trivially, and needs no protocol upgrade handling. Client→server traffic (toggle, claim) stays on the existing REST endpoints. Native `EventSource` cannot set the `X-Tenant-ID` header required by the 006 tenant-context contract, so the client wraps `fetch` with `credentials: 'include'`, parses the `text/event-stream` body, and exposes an RxJS Observable with exponential-backoff retry — which also satisfies the constitution's RxJS-first rule.
- **Alternatives considered**: **WebSocket** — bidirectional capability we don't need, can't set custom headers from the browser either, and bypasses parts of the HTTP middleware stack (separate auth handling on upgrade). **Short polling** — cannot meet SC-008 (<5 s) without aggressive intervals, gives no presence signal for FR-017a, and violates the Principle X preference for streaming. **Native EventSource with tenant in a query param** — moves tenant context out of the audited 006 header contract for one endpoint; rejected for contract consistency.

## R2. Fan-out and presence: in-process registry, DB as availability truth

- **Decision**: The escalations module keeps a process-local registry keyed by `(tenant_id, membership_id)`: a `tokio::sync::broadcast` channel per tenant for event fan-out plus a connection count per member for presence. Availability (`available`/`away`) persists in Postgres (`agent_availability`) as the manual toggle state; **routing eligibility = DB row says `available` AND presence registry shows ≥1 live stream**. When a member's last stream closes, a grace timer (~45 s, covers refreshes and brief drops) fires and writes `away` to the DB (FR-017a), emitting the change like a manual toggle. On server startup, a sweep reverts any stale `available` rows older than the grace window, so a restart never leaves phantom-available agents.
- **Rationale**: Single-process modular monolith — an in-process registry delivers sub-second FR-025 latency with zero new infrastructure, and hiding it behind the `escalations::presence` interface keeps the future extraction path (Redis pub/sub + presence keys with TTL) an implementation swap, not an API change. Keeping the toggle state in Postgres keeps the UI truthful across restarts and lets routing queries stay in SQL.
- **Alternatives considered**: **Redis pub/sub + TTL presence keys now** — operationally premature for one process (recorded as the extraction path in Complexity Tracking). **Availability purely in memory** — lost on restart and invisible to SQL routing queries. **Heartbeat rows in Postgres** — write amplification (a row touch per agent per interval) for something the connection lifecycle already tells us.

## R3. Routing concurrency: per-tenant advisory lock + single ranked selection statement

- **Decision**: Every routing decision (new escalation, queue drain pass) runs inside a transaction that first takes `pg_advisory_xact_lock(hash(tenant_id))`, then executes one candidate-selection statement: eligible members (active membership, agent-capable role, `available`, present) ranked by `count(matched required skills) DESC, load ASC, membership_id ASC` where load = open/pending conversations currently assigned (FR-007). Assignment, escalation-state update, `conversations.escalated_at` flag, audit row, and outbox/broadcast notification commit atomically. Manual claims do **not** take the tenant lock; they are a compare-and-set `UPDATE escalations SET status='assigned', … WHERE id=$1 AND tenant_id=$2 AND status='queued'` — zero rows affected → `409 conflict` with the winning assignee in the payload (US2 scenario 3).
- **Rationale**: FR-011 makes single-assignment under concurrency a hard requirement. The advisory lock serializes the *decisions* per tenant (cheap: decisions are milliseconds; contention only within one tenant) which also cleanly implements the one-at-a-time drain rule — each drain iteration re-reads load inside the lock, so a newly-online agent can't be flooded (spec edge case). CAS claims stay lock-free so a stuck routing pass can never block an agent's manual claim. Deterministic final tie-break (membership id) makes FR-024's branch tests reproducible.
- **Alternatives considered**: **`SELECT … FOR UPDATE SKIP LOCKED` on queue rows only** — protects the queue entry but not the load reads, so two concurrent escalations could both count the same agent as least-loaded and flood them. **Serializable isolation** — retry loops throughout for a problem the advisory lock solves without retries. **A background routing worker with a job table** — adds latency against SC-003 and machinery the in-request decision doesn't need; drain triggers (R4) already cover the async cases.

## R4. Queue drain triggers: event-driven, never polled

- **Decision**: A drain pass (skill-aware, per Q2 clarification: oldest entry whose required skills the agent matches, else oldest outright) is triggered by exactly the events that change eligibility or capacity: (a) an agent toggles to `available`; (b) an agent's presence is established while their DB state is `available` (reconnect); (c) an assignment ends or load drops — conversation resolved/closed or reassigned away (consumed from the outbox, R5); (d) an escalation is queued while eligible agents exist (races the no-agent check). Each pass assigns at most one conversation per free agent, then re-evaluates.
- **Rationale**: Matches FR-014 exactly and keeps the system quiet when nothing changes (Principle X). Every trigger already passes through escalations-module code or its outbox consumer, so no scheduler is needed.
- **Alternatives considered**: **Periodic sweep (e.g., every 30 s)** — adds up to a sweep-interval of avoidable queue latency and constant idle work; kept in the back pocket as a safety net if event coverage ever proves leaky, but not shipped.

## R5. Module boundary: one-way calls in, outbox events back

- **Decision**: `escalations` depends on `conversations` and calls two new public tx interfaces — `assign_in_tx(tenant, conversation, membership)` (reusing 013's assignment + audit semantics) and `set_escalated_in_tx(tenant, conversation, Option<timestamp>)` (maintains `conversations.escalated_at`, which the inbox `escalated` filter reads). In the other direction, `conversations` emits transactional-outbox events (0002 pattern, new event types `conversation.status_changed` and `conversation.assignment_changed`, carrying an `origin` field). The escalations consumer closes out queued/assigned escalations when their conversation resolves/closes (FR-015), relabels the routing reason `manual_reassignment` when a member reassigns an escalated conversation outside the routing engine (FR-023), and triggers drain on load-freeing changes. Events with `origin=escalations` (produced by routing's own `assign_in_tx` calls) are ignored by the consumer to prevent feedback loops.
- **Rationale**: Constitution I forbids cross-module data access and circular dependencies; this is the standard one-way-calls + domain-events resolution, and the outbox infrastructure (0002/0022 with delivery claims) already exists and is proven by the invitation-delivery feature. Close-out latency of an outbox tick is acceptable for FR-015 (queue hygiene), while the latency-sensitive path (routing → assignment → notification) is all synchronous inside escalations.
- **Alternatives considered**: **conversations importing escalations for the escalated filter/reassignment hooks** — circular dependency, rejected outright. **Inbox query joining the escalations table** — cross-module table access, rejected (hence the `escalated_at` flag column owned by conversations). **A new in-process synchronous event bus** — new kernel machinery duplicating what the outbox already provides, and it would couple module lifetimes; rejected.

## R6. Escalation invoker: internal application service is the contract, guarded HTTP endpoint for now

- **Decision**: The canonical escalation contract is the escalations crate's public application-service function (`escalate(tenant, conversation, reason, required_skill_ids)`) — this is the "human escalation" tool Principle IV requires the future AI subsystem to call. Until that subsystem exists (spec assumption), the same function is exposed as `POST /tenant/conversations/{id}/escalate` guarded by `conversations.manage`, documented in the contract as the AI integration point and the test surface for the "AI can escalate" acceptance criterion. Duplicate escalation of an already-active escalation → `409`.
- **Rationale**: Keeps the AI-facing contract tool-mediated and provider-independent while making the feature fully testable end-to-end today; the HTTP route costs nothing to keep later (a human-triggered escalate action is a plausible future UI affordance and harms no requirement now).
- **Alternatives considered**: **No HTTP endpoint (service-only)** — leaves the E2E path and quickstart untestable without the AI feature. **A dev-only header-gated route** — the guarded route is strictly simpler and already audit-covered.

## R7. Skills data: live links + name snapshot on the escalation record

- **Decision**: Live skill matching joins `escalations.required_skill_ids` against `agent_skills`. Each escalation additionally denormalizes `required_skill_names TEXT[]` at creation time. Skill deletion hard-deletes catalog row and links and strips the id from *queued* escalations' `required_skill_ids` (FR-019) — but never touches `required_skill_names`, so history/banner display survives deletion. Skill names are unique per tenant case-insensitively (functional unique index on `lower(name)`), 1–50 chars.
- **Rationale**: FR-019's two halves (immediate removal from routing; history retained) map cleanly onto id-links vs name-snapshot. Case-insensitive uniqueness prevents "Billing"/"billing" split catalogs — consistent with how tenant slugs are handled.
- **Alternatives considered**: **Soft-deleting skills** — queued entries would keep matching a "deleted" skill or need the same strip anyway; more states for no benefit. **Snapshot-only (no id links)** — string matching against renameable catalog entries breaks when a skill is renamed; ids stay stable.

## R8. Permissions: reuse existing codes, no matrix change

- **Decision**: No new permission codes. Queue view → `conversations.view`; escalate (interim), claim, availability toggle (self-only, enforced in-handler) → `conversations.manage`; skills catalog + per-agent skill assignment → `members.manage` (view via `members.view`); SSE stream → `conversations.view` (its events are all conversation-domain). Agent-capable routing targets = memberships whose role grants `conversations.manage` (Owner/Admin/Manager/Agent — Viewer excluded), matching FR-009 exactly.
- **Rationale**: The spec's own language maps onto the 008 catalog ("conversation-management permission", "team-management permission"); 013 set the precedent that fitting the existing matrix beats growing it. Self-only availability is a data-shape rule (you are the resource), not a permission tier.
- **Alternatives considered**: **New `escalations.view/manage` codes** — would require matrix updates across roles and migrations of granted sets for no behavioral difference from the mapping above.

## R9. Browser notifications: progressive enhancement over the in-app signal

- **Decision**: The in-app indicator (topbar badge + toast via the notifications service, fed by the SSE `escalation.assigned` event) is the always-on channel satisfying FR-025. The Notification API is layered on top: permission requested on first availability toggle-on (a user gesture), notification shown only when permission is granted and the tab is not focused; clicking it focuses/navigates to the conversation. Denied/unsupported permission never blocks anything.
- **Rationale**: FR-025 wording ("where the agent has granted permission, and always an immediate in-app indicator") prescribes exactly this shape; requesting permission on toggle-on ties the browser prompt to the moment the agent opts into receiving work.
- **Alternatives considered**: **Web Push (service worker)** — delivers when the tab is closed, but a closed tab means no presence, which means auto-away and no auto-assignments (Q5) — so the case it serves cannot occur; large scope for an unreachable state.

## R10. Frontend state placement

- **Decision**: Transport in `core/realtime` (singleton `RealtimeService` Observable + `NotificationsService` signals — core layer, no feature deps). Availability state in a small `core`-level store consumed by the topbar toggle (layout may not depend on feature areas). Queue page state in a feature-local NgRx SignalStore (`features/tenant/escalations/`), subscribing to realtime events for live updates with REST refetch as reconciliation. Conversation-detail store extends with the embedded `escalation` object. Nothing new in the global NgRx `appUi` slice (availability is server-persisted per Q5 — not a client preference like theme).
- **Rationale**: Follows the spec-002 state ladder (global cross-feature → Store, feature-local → SignalStore, temporary → signal) and the layering rule that `layout/` and `core/` never import from `features/`.
- **Alternatives considered**: **Availability in the `appUi` Store slice** — it isn't client UI state; server is the source of truth and it must survive sign-out semantics (auto-away), so a core service/store wrapping the API + SSE is the correct home.
