## Load-bearing decisions locked in for this feature (read before starting any task)

These resolve real gaps/ambiguities found in plan.md/data-model.md/research.md/contracts/ during task generation. Treat them as binding, not as suggestions:

1. **Error codes**: this codebase's `kernel::ApiError` always uses generic bucket codes (`conflict`, `validation_failed`, `not_found`) ‚Äî there is no per-scenario `code` string like `escalation_active` or `already_claimed`. Do **not** add one. The contract's `409 escalation_active` / `409 already_claimed` / `409 duplicate_name` / `422 invalid_state` labels are scenario names for the docs, not literal `error.code` values ‚Äî each route only has one possible 409/422 meaning, so the endpoint alone disambiguates for the frontend. Distinguishing payload data (e.g. the winning claimant's membership id) goes in `error.details` as a single JSON object, mirroring `customers::routes::build_identifier_conflict` (`crates/modules/customers/src/routes.rs`) ‚Äî attempt the write, catch the unique-violation, then run a follow-up `SELECT` to build a rich conflict response. Do not pre-check-then-insert (TOCTOU race); follow the attempt-then-explain pattern used there.
2. **Router state type is `PgPool`, not `AppState`**: `crates/server/src/router.rs`'s `ProtectedRoutes`/`tenant_routes()` build `Router<sqlx::PgPool>` ‚Äî handlers extract `State(pool): State<sqlx::PgPool>`. Any extra runtime object a handler needs (the escalations broadcast/presence registry) **cannot** be added as router `State` ‚Äî it must be injected as an `Extension` layer in `api_routes()`, exactly like `Extension(state.config.clone())` and `Extension(email_sender)` already are. Handlers extract it as `Extension<Arc<escalations::events::Runtime>>`.
3. **Composite FK targets that already exist ‚Äî reuse, do not recreate**: `tenant_memberships (tenant_id, id)` already has a `UNIQUE` constraint named `tenant_memberships_tenant_id_id_uq` (added in migration 0033). `agent_skills`, `agent_availability`, and `escalations.assigned_membership_id` FK against it directly ‚Äî do not re-add this constraint. `conversations (tenant_id, id)` has **no** such constraint yet ‚Äî migration 0037 must add one (name it `conversations_tenant_id_id_uq`, mirroring the exact `DROP INDEX IF EXISTS ‚Ä¶; ALTER TABLE ‚Ä¶ ADD CONSTRAINT ‚Ä¶ UNIQUE (tenant_id, id);` idiom used for `tenant_memberships` in `backend/migrations/0033_conversation_core.sql`) before `escalations.conversation_id` can carry a composite FK to it.
4. **Arrays cannot carry a `FOREIGN KEY` in Postgres**: `escalations.required_skill_ids UUID[]` has no DB-level referential integrity. Validity (ids must exist in the tenant's skill catalog) is checked in application code before insert. Skill deletion strips the id from *queued* escalations' `required_skill_ids` via an explicit `array_remove(required_skill_ids, $skill_id)` `UPDATE ‚Ä¶ WHERE status = 'queued'` run in the same transaction as the `DELETE FROM skills` ‚Äî there is no cascade to write.
5. **`matched_skill_names` snapshot (addition to data-model.md)**: data-model.md's `escalations` table lists `matched_skill_ids UUID[]` but not a name snapshot. Since a matched skill can later be renamed or deleted, and history must stay displayable (same reasoning R7 already applied to `required_skill_names`), migration 0037 also adds `matched_skill_names TEXT[] NOT NULL DEFAULT '{}'`, populated at assignment time alongside `matched_skill_ids`, only for `routing_reason IN ('skill_match','queue_auto')`. The `matchedSkills` field in API responses reads this snapshot directly ‚Äî never a live join.
6. **Routing eligibility is DB state AND in-process presence** (data-model.md / events.md / research R2, restated precisely because it changes how the ranking query must be built): an agent is a routing candidate only if `agent_availability.state = 'available'` **and** they currently hold ‚â•1 live `/tenant/events` connection on *this* server process. Presence lives only in memory ‚Äî SQL cannot see it. So candidate-selection must: (a) snapshot the currently-connected membership ids for the tenant from the in-process presence registry into a `Vec<Uuid>` *before* opening the routing transaction, then (b) pass that `Vec<Uuid>` into the ranked SQL query as `tm.id = ANY($present_ids)`. Never try to query presence from SQL.
7. **Presence must be an RAII guard tied to the SSE stream's lifetime**, not a connection-close callback wired to a specific transport event ‚Äî because integration tests establish "presence" by literally issuing a real `GET /tenant/events` request through the router (via `tower::ServiceExt::oneshot`) and holding the resulting `Response`'s body alive in a local variable for the duration of the assertion; there is no real TCP close to hook in that setup. Implement a `PresenceGuard` (`(tenant_id, membership_id, registry handle)`) whose `Drop` decrements the live-connection count and, if it reaches zero, spawns a grace-window timer that reverts the DB row to `away` if still zero when it fires. The SSE handler's response stream must own one `PresenceGuard` for its whole lifetime (wrap the broadcast-subscriber stream in a small struct that also holds the guard, so dropping the stream ‚Äî client disconnect, or a test dropping the `Response` ‚Äî drops the guard).
8. **Presence grace window is a constructor parameter, not a hardcoded constant**: production defaults to 45s (research R2), but integration tests need this short (they cannot wait 45 real seconds). The escalations `Runtime`/presence registry takes a `grace: Duration` at construction. `main.rs` constructs it with 45s; test harnesses (`crates/server/tests/escalations.rs`, and any other test building `AppState`) construct it with a short duration (e.g. 1s) so presence-revert assertions run fast.
9. **Startup sweep = schedule the same grace timer for every already-`available` row, not a one-shot age-filtered `UPDATE`.** A one-shot `UPDATE ‚Ä¶ WHERE state='available' AND state_changed_at < now() - grace` leaves a real gap: a row toggled available in the moments just before restart would be younger than the grace window and would never revert if its owner never reconnects (no timer was ever scheduled for it in the new process). Instead, at boot, for every row currently `state = 'available'` in `agent_availability`, call the exact same "presence lost, start grace timer" function used for a live disconnect, as if that row's last connection had just closed. This reuses one code path and closes the gap.
10. **Advisory lock key**: use the two-key form to namespace against any future advisory-lock user: `SELECT pg_advisory_xact_lock(hashtext('escalations.routing'), hashtext($1))` bound to `tenant_id.to_string()`. Do not attempt to cast a UUID into a single bigint key.
11. **Candidate-selection is one ranked SQL statement** combining skill-match count, load count, and the presence filter from decision 6, e.g. (adjust column aliases as needed, but preserve the shape and ordering exactly):
    ```sql
    SELECT tm.id AS membership_id,
           COALESCE(m.match_count, 0) AS match_count,
           COALESCE(m.matched_ids, '{}') AS matched_skill_ids,
           COALESCE(l.load_count, 0) AS load_count
    FROM tenant_memberships tm
    JOIN agent_availability aa ON aa.tenant_id = tm.tenant_id AND aa.membership_id = tm.id
    LEFT JOIN LATERAL (
        SELECT COUNT(*) AS match_count, array_agg(ask.skill_id) AS matched_ids
        FROM agent_skills ask
        WHERE ask.tenant_id = tm.tenant_id AND ask.membership_id = tm.id
          AND ask.skill_id = ANY($required_skill_ids)
    ) m ON true
    LEFT JOIN LATERAL (
        SELECT COUNT(*) AS load_count FROM conversations c
        WHERE c.tenant_id = tm.tenant_id AND c.assigned_membership_id = tm.id
          AND c.status IN ('open','pending') AND c.deleted_at IS NULL
    ) l ON true
    WHERE tm.tenant_id = $tenant_id AND tm.status = 'active' AND tm.deleted_at IS NULL
      AND tm.role IN ('owner','admin','manager','agent')
      AND aa.state = 'available'
      AND tm.id = ANY($present_ids)
    ORDER BY match_count DESC, load_count ASC, tm.id ASC
    LIMIT 1
    ```
    When `$required_skill_ids` is empty, `match_count` is 0 for every row, so ordering degrades to pure load-based selection automatically (FR-006) ‚Äî this is intentional, do not special-case "no skills required".
12. **"Agent-capable role"** (FR-009) is the literal SQL list `role IN ('owner','admin','manager','agent')` ‚Äî confirmed against `crates/modules/authz/src/matrix.rs`: `TENANT_ADMIN`/`TENANT_MANAGER`/`TENANT_AGENT` all include `Permission::ConversationsManage`; `TENANT_VIEWER` does not. Do not derive this from a Rust-side permission check inside a per-row loop ‚Äî hardcode the role list in SQL.
13. **Outbox event emission** reuses the existing free-text `outbox_events.event_type` column (no CHECK constraint restricts it ‚Äî confirmed in `backend/migrations/0002_outbox.sql`/`0022_outbox_delivery_claims.sql`). Migration 0037 adds one dedicated partial index for the two new event types, mirroring `outbox_invitation_delivery_claimable_idx`'s exact shape.
14. **Escalations JSON is camelCase-native**, mirroring `tenancy::members::TeamMemberResponse`'s `#[serde(rename_all = "camelCase")]` convention ‚Äî **not** the `conversations` module's snake_case-plus-frontend-mapper convention. Every escalations `model.rs` response struct gets `#[serde(rename_all = "camelCase")]`. Consequence: the frontend `Escalation`/`QueueEntry`/`Skill`/`Availability` TypeScript interfaces are used directly as the HTTP response shape ‚Äî **no `xFromWire`/`xToWire` mapping functions are needed or wanted for these new types.** Do not write them.
15. **Cross-module response composition must live in the `server` crate, never as a new cross-module Rust dependency.** `escalations` already depends on both `conversations` and `tenancy` (plan.md). If `conversations` or `tenancy` imported `escalations` to embed its data, Cargo would reject it outright as a cyclic package dependency ‚Äî this is not a style preference, it will not compile. Two response shapes need data from both sides (conversation-detail + its escalation; member roster + skills/availability) ‚Äî both are composed by **new handler functions living in `crates/server/src/`**, which already depends on every module crate. This exact pattern is already precedented by the `login` handler in `crates/server/src/router.rs`, which composes `identity::routes::authenticate_login` and `tenancy::routes::build_me_response` directly. Do not attempt to have `conversations` or `tenancy` call into `escalations`.
16. **Drain is one attempt per trigger, never a loop.** "Toggle to available" / "presence reconnect while available" / "an assignment's load frees up" each cause **exactly one** `drain_one_for_membership_in_tx` call for the one affected agent ‚Äî not a `while` loop draining the whole queue onto them. This is what the spec's one-at-a-time edge case and the quickstart's "exactly one auto-assignment per capacity re-evaluation" check require. The remaining queued entries wait for the *next* trigger (which may be the same agent again once their newly-assigned conversation resolves and frees their load).
17. **The "escalation queued while an eligible agent already exists" race (research R4 trigger d) needs no dedicated handler.** Because every routing decision ‚Äî new-escalation and drain alike ‚Äî takes the same per-tenant advisory lock, a new escalation and a concurrent toggle/reconnect can never interleave: whichever transaction commits first is what the other one's candidate query (or the next drain trigger) sees. Do not write extra "recheck queue after escalating" code.
