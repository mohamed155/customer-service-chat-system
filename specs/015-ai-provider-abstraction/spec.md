# Feature Specification: AI Provider Abstraction

**Feature Branch**: `015-ai-provider-abstraction`

**Created**: 2026-07-15

**Status**: Draft

**Input**: User description: "AI Provider Abstraction — Create a provider-independent AI layer. Scope: AI provider interface, OpenAI adapter, Anthropic adapter, Gemini adapter, model configuration, API key storage, provider selection per tenant, token usage tracking. Backend: AI provider trait/interface, provider registry, chat completion abstraction, streaming abstraction where possible, provider configuration model. Acceptance: backend can call at least one real AI provider, provider can be switched by configuration, token usage is recorded, LLM logic does not leak into business modules."

## Clarifications

### Session 2026-07-15

- Q: Whose vendor API keys do tenants use? → A: Hybrid — the platform supplies default keys per provider; a tenant configuration MAY carry its own key (BYOK), which fully replaces the platform key for that tenant. A tenant configuration without its own key uses the platform key for its chosen provider; if neither exists, AI is not configured for that tenant.
- Q: What does the AI layer do on provider failure? → A: Automatic failover — transient failures (rate-limited, unavailable, timeout) get a small bounded retry with backoff on the same provider, then the layer fails over through the configuration's ordered fallback providers; the normalized error surfaces only after retries and all fallbacks are exhausted. The response and the usage record always identify the provider/model that actually served the call.
- Q: Does the AI layer persist prompt/reply content alongside usage records? → A: Configurable per tenant — content capture is off by default (metadata-only usage records); a tenant can opt in, after which request messages and reply content are persisted with the usage record, tenant-isolated and access-restricted. Content never appears in logs or traces regardless of the setting, and toggling the setting is audited.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - The Platform Generates an AI Response Through the Configured Provider (Priority: P1)

The platform needs an AI-generated chat reply — for example, a future AI runtime composing an answer to a customer message. The requesting module hands the AI layer a conversation-style request (system instructions plus a sequence of prior messages) together with the tenant it acts for. The AI layer resolves which provider and model that tenant is configured to use, sends the request to that vendor with the stored credentials, and returns the assistant's reply in a single uniform shape — the same shape regardless of which vendor produced it. The requesting module never knows or cares which vendor was used.

**Why this priority**: This is the entire reason the feature exists — a working, provider-independent path from "the platform wants an AI reply" to "a real AI vendor produced one". Every other story configures, secures, or measures this path.

**Independent Test**: With one vendor configured with a valid key, invoke the AI layer with a sample conversation and verify a real completion comes back in the uniform response shape, with the vendor-specific details fully hidden from the caller. Deliverable value: the platform can produce real AI output today.

**Acceptance Scenarios**:

1. **Given** a tenant configured to use a specific provider and model with a valid credential, **When** a module requests a chat completion for that tenant, **Then** a reply generated by that provider's model is returned in the platform's uniform response shape.
2. **Given** the same request payload, **When** it is sent to tenants configured with different providers, **Then** each tenant's configured vendor handles it and the responses arrive in the identical uniform shape.
3. **Given** a request for a tenant, **When** the AI layer calls the vendor, **Then** the vendor call uses only that tenant's resolved configuration (provider, model, parameters, credential) — never another tenant's.
4. **Given** a completed AI call, **When** the caller inspects the result, **Then** it contains the reply content, the model that produced it, and the token usage consumed, without exposing vendor-specific wire formats.
5. **Given** the primary provider fails transiently (rate limit, outage, timeout) and the configuration defines a fallback provider, **When** a completion is requested, **Then** the layer retries briefly, then transparently serves the call through the fallback, and the response and usage record identify the fallback provider/model as the one that served it.
6. **Given** every configured provider (primary, then fallbacks in order) fails or no fallback is defined, **When** the AI layer gives up, **Then** the caller receives a normalized error category (authentication, rate-limited, unavailable, timeout, invalid request) rather than a raw vendor error.

### User Story 2 - An Administrator Switches Providers by Configuration Alone (Priority: P2)

An authorized administrator decides a tenant should use a different AI vendor or model — for cost, quality, or compliance reasons. They update the tenant's AI configuration (provider, model, generation parameters). From the next AI request onward, the platform uses the new provider; no code change, redeploy, or involvement from engineering is needed, and modules consuming AI notice nothing except (possibly) different answer style.

**Why this priority**: Provider independence is only real if switching is an operational action, not an engineering project. It is also the guard against vendor lock-in demanded by the platform's principles.

**Independent Test**: Configure a tenant on provider A, make a request, switch the configuration to provider B, make the same request, and verify each request was served by the then-configured provider with no other part of the system touched.

**Acceptance Scenarios**:

1. **Given** a tenant configured for provider A, **When** an authorized administrator changes the tenant's configuration to provider B with a valid model and credential, **Then** subsequent AI requests for that tenant are served by provider B.
2. **Given** a configuration change, **When** it is saved, **Then** the change is recorded in the audit trail with who changed what and when.
3. **Given** a member without permission to manage AI configuration, **When** they attempt to view or change it, **Then** the system refuses.
4. **Given** a tenant with no AI configuration of its own, **When** an AI request is made for it, **Then** the platform-level default configuration is used; if no default exists either, the request fails with a clear "not configured" error rather than a crash.
5. **Given** an administrator updating a configuration, **When** they select a provider, **Then** they can set the model identifier and generation parameters (such as maximum reply length and creativity/temperature) that all subsequent requests for that tenant will use.

---

### User Story 3 - API Keys Are Stored and Handled Securely (Priority: P3)

An authorized administrator enters an AI vendor API key for the platform or for a specific tenant. The key is stored encrypted, is never shown back in full after entry (only a masked hint such as the last few characters), never appears in logs or error messages, and can be replaced or removed at any time. Rotating a key takes effect on the next AI request.

**Why this priority**: Real vendor calls (P1) need credentials, and credentials are the most sensitive data this feature introduces; mishandling them is a security incident. It ranks after P1/P2 only because those stories define the flows the keys unlock.

**Independent Test**: Enter a key, verify it is accepted and usable for AI calls but never retrievable in plaintext through any interface or log; replace it and verify the new key is used; remove it and verify AI calls fail with an authentication/not-configured error.

**Acceptance Scenarios**:

1. **Given** an authorized administrator, **When** they save an API key for a provider, **Then** the key is stored encrypted at rest and subsequent AI calls using that configuration authenticate with it.
2. **Given** a stored API key, **When** any user or system reads the configuration back, **Then** only a masked identifier (e.g., last four characters) is ever returned — the full key is never retrievable through any interface.
3. **Given** a stored API key, **When** AI calls succeed or fail, **Then** the key never appears in logs, traces, audit entries, or error messages.
4. **Given** an existing key, **When** an administrator replaces (rotates) or deletes it, **Then** the change takes effect for the next AI request and the rotation/deletion is audited (without recording the key material itself).
5. **Given** an invalid or revoked key, **When** an AI request uses it, **Then** the caller receives the normalized authentication error and the failure is visible in observability output without leaking the key.

---

### User Story 4 - Every AI Call's Token Usage Is Recorded (Priority: P4)

Every AI request the platform makes — successful or failed after reaching the vendor — produces a usage record: which tenant, which provider and model, how many input and output tokens, when, and whether it succeeded. Authorized users can retrieve usage figures for a tenant over a time period, giving the business the raw data for cost monitoring and future billing.

**Why this priority**: Token spend is real money per call; untracked usage is unbounded cost risk. It depends on P1 existing but must land in the same feature so no untracked calls ever ship.

**Independent Test**: Make several AI calls across tenants (including a failing one), then verify each produced a usage record with correct attribution and token counts, and that per-tenant totals over a period can be retrieved.

**Acceptance Scenarios**:

1. **Given** a successful AI call, **When** it completes, **Then** a usage record is stored with the tenant, provider, model, input tokens, output tokens, timestamp, and success status.
2. **Given** an AI call that reaches the vendor but fails or is cut short, **When** it ends, **Then** a usage record is still stored capturing whatever usage occurred and the failure status.
3. **Given** usage records across multiple tenants, **When** usage for one tenant is queried for a period, **Then** only that tenant's records are included and totals (calls, input tokens, output tokens) are correct.
4. **Given** an unauthorized member, **When** they attempt to read usage data, **Then** the system refuses.
5. **Given** a tenant with content capture at its default (off), **When** AI calls complete, **Then** usage records contain metadata and counts only — no prompt or reply content is persisted.
6. **Given** a tenant that has opted into content capture, **When** an AI call completes, **Then** the request messages and reply content are persisted with the usage record, retrievable only by authorized roles of that tenant, and the opt-in change itself is audited.

---

### User Story 5 - AI Replies Can Stream Incrementally (Priority: P5)

When a consuming module asks for a streamed completion and the configured provider supports streaming, the AI layer delivers the reply incrementally as the vendor produces it, ending with the same final metadata (model, token usage) as a non-streamed call. If the provider or configuration cannot stream, the layer transparently falls back to delivering the complete reply at once through the same interface.

**Why this priority**: Streaming is a latency-experience improvement mandated by the platform's performance principles, but a correct non-streaming path (P1) already delivers the feature's core value.

**Independent Test**: Request a streamed completion from a streaming-capable provider and verify content arrives in multiple increments before completion, with final usage recorded; request the same from a non-streaming path and verify the caller still receives the full reply via the same interface.

**Acceptance Scenarios**:

1. **Given** a streaming-capable provider, **When** a module requests a streamed completion, **Then** reply content is delivered in increments as generated, followed by completion metadata including token usage.
2. **Given** a streamed call, **When** it completes or fails mid-stream, **Then** a usage record is stored just as for non-streamed calls, reflecting what was consumed.
3. **Given** a provider or configuration that cannot stream, **When** a module requests a streamed completion, **Then** the module still receives the full reply through the same interface (delivered as a single increment) rather than an error.
4. **Given** a stream interrupted by a vendor failure, **When** the interruption occurs, **Then** the caller receives a normalized error signal on the stream and the partial usage is recorded.

---

### Edge Cases

- What happens when a tenant has no AI configuration and no platform default exists? The request fails fast with a clear "AI not configured" error; nothing is sent to any vendor and no usage record is created.
- What happens when the configured model identifier is invalid or has been retired by the vendor? The vendor's rejection is normalized to an invalid-request error; the configuration remains editable so an administrator can correct the model.
- What happens when the vendor rate-limits or is down? The layer retries briefly, then fails over to the next configured fallback provider; the caller sees the normalized rate-limited/unavailable error only when no fallback remains. Every attempt is observable (logs/traces/metrics) and attributable to its provider, and any usage a vendor reported is still recorded.
- What happens when a fallback provider serves the call? The response and usage record attribute the actual provider/model used (not the primary), so cost and behavior remain auditable.
- What happens when a streamed reply fails after content has already been delivered? No mid-stream provider switch occurs — failover applies only before the first increment is delivered; a stream that dies mid-reply surfaces the normalized error on the stream and records partial usage.
- What happens when configuration changes while a request is in flight? The in-flight request completes under the configuration it started with; the new configuration applies from the next request.
- What happens when a vendor response omits usage figures? The usage record is stored with the counts marked as unreported rather than silently recorded as zero consumption.
- What happens when two tenants use the same vendor with different keys? Each call authenticates with its own tenant's resolved credential; usage is attributed per tenant.
- What happens when a request would exceed the configured maximum reply length? The reply is truncated by the vendor per the configured limit and returned with its usage; this is a normal completion, not an error.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST expose a single uniform chat-completion capability (conversation-style request in, assistant reply out) that all consuming modules use for AI generation; consuming modules MUST NOT contain vendor-specific logic, vendor names, or vendor wire formats.
- **FR-002**: The system MUST support OpenAI, Anthropic, and Gemini as interchangeable providers behind that uniform capability.
- **FR-003**: Adding a further provider MUST require only implementing the provider-facing contract — no changes to any consuming module.
- **FR-004**: The system MUST resolve, per request, which provider/model/parameters to use from the acting tenant's AI configuration, falling back to the platform-level default configuration when the tenant has none, and failing with a clear "not configured" error when neither exists. The credential resolves independently: the tenant's own key (BYOK) when present, otherwise the platform key for the chosen provider, otherwise the same "not configured" failure.
- **FR-005**: Authorized administrators MUST be able to create and update AI configurations: primary provider choice, model identifier, generation parameters (at minimum maximum reply length and temperature/creativity), and an optional ordered list of fallback provider/model entries used for failover.
- **FR-006**: Provider selection and configuration changes MUST take effect from the next AI request without redeployment or code changes.
- **FR-007**: All AI configuration reads and writes MUST be restricted to authorized roles, and every configuration change (including key rotation/deletion) MUST be audited with who/what/when — excluding secret material.
- **FR-008**: API keys MUST be stored encrypted at rest, MUST never appear in source control, logs, traces, audit records, or error messages, and MUST never be retrievable in full after entry — only a masked hint may be shown.
- **FR-009**: Administrators MUST be able to set, rotate, and delete API keys per configuration, with changes effective on the next request.
- **FR-010**: Every AI call that reaches a vendor MUST produce a usage record containing tenant, provider, model, input tokens, output tokens (marked unreported when the vendor omits them), timestamp, and success/failure status — for streamed and non-streamed calls alike.
- **FR-011**: Authorized users MUST be able to retrieve a tenant's usage records and period totals; tenant isolation MUST apply to both configurations and usage records.
- **FR-012**: Vendor errors MUST be normalized into uniform categories (at minimum: authentication failure, rate-limited, provider unavailable, timeout, invalid request) so consuming modules handle failures identically across vendors; a normalized error is surfaced only after retries and all configured fallbacks are exhausted.
- **FR-013**: The system MUST support incremental (streamed) delivery of replies through the uniform capability where the provider supports it, and MUST transparently deliver the complete reply through the same interface where it does not.
- **FR-014**: Administrators MUST be able to verify a configuration's connectivity (a test invocation confirming the provider accepts the credential and model) without wiring AI into any customer-facing flow.
- **FR-015**: AI calls MUST participate in the platform's observability: request-ID propagation, structured logs, traces, and latency/error visibility per provider — without exposing prompt secrets or credentials.
- **FR-016**: The AI layer MUST NOT read or write business data on its own; it operates solely on what callers pass in, preserving the platform rule that models never access internal services or databases directly.
- **FR-017**: On transient failures (rate-limited, provider unavailable, timeout) the AI layer MUST retry the same provider a small bounded number of times with backoff, then automatically fail over through the configuration's ordered fallback providers (each with its own resolved credential); non-transient failures (authentication, invalid request) MUST NOT trigger retry or failover. Every retry and failover MUST be observable, and the result and usage record MUST identify the provider/model that actually served the call.
- **FR-018**: Content capture MUST be a per-tenant setting, off by default. When off, no prompt or reply content is persisted by the AI layer; when a tenant opts in, request messages and reply content (including partial replies from interrupted streams) are persisted alongside the usage record, tenant-isolated and readable only by authorized roles. Toggling the setting MUST be audited and MUST apply from the next AI call. Prompt and reply content MUST NOT appear in logs or traces regardless of this setting.

### Key Entities

- **AI Provider**: One of the supported vendors (OpenAI, Anthropic, Gemini) — a fixed catalog entry describing what the platform can talk to, including whether it supports streaming.
- **AI Configuration**: A primary provider, model identifier, and generation parameters, plus an optional ordered list of fallback provider/model entries for failover; exists at platform level (default) or per tenant (override); the unit an administrator edits and audits.
- **API Credential**: A vendor secret stored encrypted and referenced by a masked hint, rotatable and deletable; exists at platform level (the default key per provider) or at tenant level (a BYOK key that fully replaces the platform key for that tenant).
- **Token Usage Record**: An append-only record of one AI call — tenant, provider/model that actually served it, input/output token counts, timestamp, success/failure — the raw material for cost monitoring and future billing. Carries the request and reply content only for tenants that have opted into content capture.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A live end-to-end AI completion succeeds against at least one real vendor using stored configuration and credentials.
- **SC-002**: Switching a tenant between any two of the three supported providers is a pure configuration action — zero code changes, zero redeploys — and takes effect on the very next AI request.
- **SC-003**: 100% of AI calls that reach a vendor (success or failure, streamed or not) produce a correctly attributed usage record; per-tenant period totals match the sum of individual records.
- **SC-004**: After entry, an API key is never observable in plaintext through any interface, log, trace, or error output.
- **SC-005**: No module outside the AI layer contains vendor-specific AI logic; a simulated fourth provider can be added without modifying any consuming module.
- **SC-006**: Streamed replies begin arriving at the consumer before generation completes (first increment delivered while the vendor is still producing), rather than after the full reply is ready.
- **SC-007**: All configuration changes and key rotations performed during acceptance testing appear in the audit trail with actor and timestamp.
- **SC-008**: With a fallback configured, a simulated transient outage of the primary provider does not fail the AI call — it is served by the fallback, attributed to the fallback in the usage record, and the failover is visible in observability output.

## Assumptions

- This feature delivers the callable AI layer, its configuration, and a connectivity verification action; actually wiring AI replies into customer conversations (the AI runtime: prompts, tools, RAG, escalation triggers) is a later feature that will consume this layer.
- Administration of AI configurations, keys, and usage data happens through the platform's API in this feature (audited and role-guarded); dashboard screens for AI settings and usage analytics are out of scope and may arrive in a later spec.
- Configurations exist at two levels: a platform-level default and optional per-tenant overrides. Keys are hybrid: the platform holds default keys per provider, and a tenant configuration may carry its own key (BYOK) that fully replaces the platform key for that tenant; a tenant configuration without its own key rides on the platform key for its chosen provider.
- "Chat completion" (system instructions plus message history in, assistant reply out) is the abstraction's scope for this feature; embeddings, tool calling, and multimodal inputs are future extensions the contract should not preclude but need not implement now.
- Usage records capture raw consumption (plus opt-in content capture per FR-018); quotas, spending limits, pricing, billing, and content retention/expiry policies are out of scope.
- Managing AI configuration is a sensitive operation reserved for tenant Owner/Admin roles (and the equivalent platform roles for platform-level defaults), consistent with the existing RBAC model.
- Vendor API keys are supplied by the business or the tenant; obtaining vendor accounts is outside the system's responsibility.
