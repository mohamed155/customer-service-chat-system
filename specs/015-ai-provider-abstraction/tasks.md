## Load-bearing decisions locked in for this feature (read before starting any task)

These resolve real gaps/ambiguities found in the design docs during task generation, verified against the current codebase. Treat them as binding:

1. **Error codes are the generic kernel buckets, never custom strings.** `kernel::ApiError` (see `backend/crates/shared/kernel/src/lib.rs`) only produces codes like `not_found`, `validation_failed`, `conflict`, `unprocessable`. The contract's `404 ai_not_configured` is a *scenario label for the docs*, not a literal `error.code` — return `ApiError::not_found("AI is not configured")`. Do **not** add new codes to kernel, and do **not** implement the contract's `ai_provider_error` 502 code anywhere: no HTTP completion endpoint exists in this feature, so that code has no producer yet.

2. **Success-response envelope idiom** (mirror `crates/modules/escalations/src/routes.rs` and `crates/modules/conversations/src/routes.rs`): single objects → `Json(json!(view))` (bare object, no wrapper); simple lists → `Json(json!({ "data": items }))`; cursor-paginated lists → a local `PaginatedResponse { data: Vec<T>, pagination: { next_cursor: Option<String>, has_more: bool } }` struct exactly like `conversations::routes`. Request bodies are extracted with `kernel::ApiJson<T>`.

3. **Router state is `sqlx::PgPool`, not `AppState`.** `crates/server/src/router.rs` builds `Router<sqlx::PgPool>`; handlers take `State(pool): State<sqlx::PgPool>`, the middleware-resolved tenant as `ctx: tenancy::TenantContext`, the actor as `Extension(principal): Extension<identity::Principal>`. The `AiService` runtime object **cannot** be router state — it is injected in `api_routes()` as `.layer(Extension(state.ai.clone()))` right next to the existing `.layer(Extension(state.escalations.clone()))`, and handlers extract `Extension(ai): Extension<ai::AiService>`. `AiService` must therefore be `Clone` (a thin wrapper around one `Arc<AiServiceInner>`).

4. **`SecretKey` lives in `ai-providers`, not `modules/ai`.** The `ChatProvider` trait takes `key: &SecretKey`, and `ai-providers` has no project dependencies, so the type must be defined in `crates/ai-providers/src/contract.rs` (plan.md places it in `crypto.rs` — that file *uses* it; the definition must sit below both in the dependency graph). Implementation: `pub struct SecretKey(String)` with `#[derive(Zeroize, ZeroizeOnDrop)]` from the `zeroize` crate, a manual `Debug` impl printing `SecretKey(****)`, **no** `Serialize`/`Display`/`Clone` derive (Clone is allowed if needed, Serialize never), and accessors `SecretKey::new(String)` + `fn expose(&self) -> &str` (the only read path).

5. **Migration conventions come from the actual migrations, not the 005 doc summary.** PKs are `UUID PRIMARY KEY DEFAULT gen_random_uuid()` (the codebase does NOT use v7 defaults — mirror `migrations/0035_agent_skills.sql`). `updated_at` maintenance is a **per-table** trigger function named `set_<table>_updated_at()` (copy the exact `CREATE OR REPLACE FUNCTION … CREATE TRIGGER …` idiom from 0035). Never edit an applied migration; the three new files are `0038_…`, `0039_…`, `0040_…` exactly.

6. **Adding `AppConfig` fields breaks every test file** — they construct `config::AppConfig { … }` as struct literals. After T005 you MUST update the `test_config()` helper in all of: `crates/server/tests/{auth,conversations,cors,customers,errors,escalations,health,live_deps,platform_tenants,rbac,team_members,tenancy,tracing}.rs` (13 files) plus the unit tests inside `crates/shared/config/src/lib.rs`. Same story for the new `AppState.ai` field (T015): update every `AppState { … }` literal in those test files (find them with `grep -rn "AppState {" backend/crates/server/tests backend/crates/server/src`).

7. **Usage records are written inline (awaited), not via a spawned task.** Research R8 proposed fire-and-forget write-behind; this feature instead awaits the single-row insert before `complete()` returns (and, for streams, before the final event is yielded). Reason: the insert is one indexed statement (single-digit ms, negligible vs. vendor latency) and inline writes make SC-003's "100% of calls produce a record" deterministic in tests with zero synchronization machinery. This is a deliberate, recorded deviation.

8. **Adapters implement `stream()` in two stages.** In US1 (T017–T019) each adapter's `stream()` is implemented as: call `self.complete(key, req).await?`, then return a stream of exactly two items — `StreamEvent::Delta(completion.content)` then `StreamEvent::Done { usage, model, finish }` (build it with `futures::stream::iter([...]).boxed()`). This is simultaneously the interim implementation and the permanent FR-013 "transparent non-streaming fallback" semantic. US5 (T040–T042) replaces the body with real SSE decoding; the two-item fallback shape moves into `AiService::stream` for the `supports_streaming() == false` case.

9. **Retry/failover policy constants** (FR-017 / research R6): per provider, 1 initial attempt + 2 retries, only when `ProviderError.retriable == true` (RateLimited, Unavailable, Timeout); sleep 200 ms before retry 1 and 1 000 ms before retry 2, each multiplied by a jitter factor drawn uniformly from `0.75..=1.25` via `rand`. Then advance to the next fallback entry (re-resolving that provider's credential) and run the same per-provider policy. `Authentication`/`InvalidRequest` abort the whole call immediately — no retry, no failover. On total failure, surface the **last** attempt's category and provider. Every attempt emits `tracing::info!` with fields `provider`, `model`, `attempt`, `outcome`/`category`, `latency_ms`, `request_id` — never message content, never key material.

10. **Config PUT is a transaction: `SELECT … FOR UPDATE` then UPDATE-or-INSERT.** Do not use `INSERT … ON CONFLICT` (the uniqueness is enforced by *partial* unique indexes, which make conflict targets fiddly). Inside one transaction: select the live row for the scope (`tenant_id = $1` or `tenant_id IS NULL`, `deleted_at IS NULL`) with `FOR UPDATE`; if found, `UPDATE` all mutable columns; else `INSERT`. Full-replace semantics: every omitted optional field becomes NULL/default (`capture_content` omitted → `false`). Audit in the same transaction via `tenancy::audit::record_in_tx`.

11. **Resolution SQL** (FR-004, one ordered lookup per table):
    ```sql
    -- configuration
    SELECT * FROM ai_configurations
    WHERE (tenant_id = $1 OR tenant_id IS NULL) AND deleted_at IS NULL
    ORDER BY tenant_id NULLS LAST LIMIT 1
    -- credential, per provider being attempted
    SELECT * FROM ai_credentials
    WHERE (tenant_id = $1 OR tenant_id IS NULL) AND provider = $2 AND deleted_at IS NULL
    ORDER BY tenant_id NULLS LAST LIMIT 1
    ```
    `capture_content` is honored **only** from a row where `tenant_id IS NOT NULL`; when the resolved config is the platform default, capture is always off. A fallback entry whose provider has no resolvable credential is **skipped** with a `tracing::warn!` event; if no attempt has a credential, the call fails `NotConfigured` (no vendor call, no usage row).

12. **Encryption specifics** (FR-008 / research R5): AES-256-GCM via the `aes-gcm` crate. Master key: env `APP_AI_KEY_ENCRYPTION_KEY`, base64 (standard alphabet) of exactly 32 bytes, decoded and validated in `AppConfig::from_env`. Per-write random 12-byte nonce from `getrandom`. AAD string: `"<tenant_uuid>|<provider>"` for tenant scope, `"platform|<provider>"` for platform scope — pass via `aes_gcm::aead::Payload { msg, aad }`. `key_hint` = last 4 characters of the plaintext key (if the key is shorter than 4 chars, the whole key — don't panic), stored as its own column and computed at write time; read paths never touch `ciphertext`. Deterministic test master key (use everywhere in tests): `"MDEyMzQ1Njc4OWFiY2RlZjAxMjM0NTY3ODlhYmNkZWY="` (base64 of the 32 ASCII bytes `0123456789abcdef0123456789abcdef`).

13. **Vendor wire details** (research R2; base URLs are `RegistryConfig` fields so tests can point at wiremock):
    | | OpenAI | Anthropic | Gemini |
    |---|---|---|---|
    | Default base URL | `https://api.openai.com` | `https://api.anthropic.com` | `https://generativelanguage.googleapis.com` |
    | Blocking path | `POST {base}/v1/chat/completions` | `POST {base}/v1/messages` | `POST {base}/v1beta/models/{model}:generateContent` |
    | Streaming | same path, body `"stream": true, "stream_options": {"include_usage": true}` | same path, body `"stream": true` | `POST {base}/v1beta/models/{model}:streamGenerateContent?alt=sse` |
    | Auth header | `Authorization: Bearer <key>` | `x-api-key: <key>` + `anthropic-version: 2023-06-01` | `x-goog-api-key: <key>` |
    | System prompt | first message, `role: "system"` | top-level `"system"` field | top-level `"systemInstruction": {"parts": [{"text": …}]}` |
    | Max tokens / temp | `max_tokens`, `temperature` | `max_tokens` (**required** by the API — send 4096 when config has none), `temperature` | `generationConfig.maxOutputTokens`, `generationConfig.temperature` |
    | Usage fields | `usage.prompt_tokens` / `usage.completion_tokens` | `usage.input_tokens` / `usage.output_tokens` | `usageMetadata.promptTokenCount` / `usageMetadata.candidatesTokenCount` |
    Error normalization (identical in all three adapters): HTTP 401/403 → `Authentication`; 429 → `RateLimited`; any 5xx or connect error → `Unavailable`; reqwest timeout → `Timeout`; any other 4xx (including unknown-model rejections) → `InvalidRequest`. `ProviderError.detail` carries the vendor's error *message text only* — never the request body, never headers. The shared `reqwest::Client` is built with `.timeout(Duration::from_secs(60)).connect_timeout(Duration::from_secs(10))`.

14. **`AiService` construction**: `AiService::from_config(pool: sqlx::PgPool, config: &config::AppConfig) -> Result<AiService, String>` builds the `ai_providers::Registry` internally from the base-URL override fields and decodes the master key — the `server` crate never imports `ai-providers` directly (SC-005 stays enforced by the dependency graph: only `modules/ai` links it). When `ai_key_encryption_key` is `None` (Test environment allows this), `from_config` still succeeds with `master_key: None`; credential seal/open then returns an internal error if ever reached — tests that exercise credentials always set the key from decision 12.

15. **The SC-005 registry seam must be a cargo feature, not `#[cfg(test)]`.** `#[cfg(test)]` items are invisible to *dependent* crates' tests. `ai-providers` declares `[features] test-providers = []` and gates `Registry::with_override(self, kind_name_or_custom: &str, provider: Arc<dyn ChatProvider>) -> Self` behind `#[cfg(feature = "test-providers")]`. `modules/ai` enables it from `[dev-dependencies] ai-providers = { path = "../../ai-providers", features = ["test-providers"] }`.

16. **`rbac.rs` expectation vectors are position-coupled to `TENANT_OPERATIONS`.** After adding entries to `TENANT_OPERATIONS`/`PLATFORM_OPERATIONS` in `crates/server/tests/rbac.rs`, every role-expectation array in that file (the tests around "Positions correspond to TENANT_OPERATIONS") must gain matching entries. The source of truth for which role holds `ai_agent.view`/`ai_agent.manage`/`platform.admin` is `crates/modules/authz/src/permission.rs` (`role_permissions`). Procedure: add the route entries, run `REQUIRE_DB_TESTS=1 cargo test -p server --test rbac`, and reconcile each failing expectation against that matrix — do not guess.

17. **Streamed `AiService::stream` implementation shape**: spawn a `tokio::task` that drives resolution → policy → vendor stream and sends `ai::AiStreamEvent`s through a `tokio::sync::mpsc::channel(32)`; return `tokio_stream::wrappers::ReceiverStream` boxed as `AiResultStream`. The driver task writes the usage record (decision 7) after forwarding `Done` or an error, *before* sending the final event, so a test that has consumed the stream to its end can immediately assert on `ai_usage_records`. Failover applies only until the first `Delta` has been sent into the channel; afterwards a vendor failure emits `AiStreamEvent::Error { category }` (recording partial usage, with partial `response_content` if capture is on) and ends the stream.

18. **Connectivity test endpoint** (FR-014): resolves config + credential exactly like a real call, then invokes the **primary** provider's `complete()` directly — no retry, no failover, no usage record. Request: `ChatRequest { system: None, messages: [Message { role: User, content: "ping" }], model: <config.model>, max_output_tokens: Some(16), temperature: None }`. Success → `200 Json(json!({"ok": true, "provider": …, "model": …, "latency_ms": …}))`; provider failure → `(StatusCode::UNPROCESSABLE_ENTITY, Json(json!({"ok": false, "error_category": <category string>, "detail": <sanitized detail>})))` (custom body per the contract, NOT `ApiError`); nothing resolvable → `ApiError::not_found("AI is not configured")`.

19. **Serialization of enums crossing the DB/API boundary**: providers serialize as `"openai" | "anthropic" | "gemini"` (`ProviderKind::as_str()`, matching the DB CHECKs), error categories as `"authentication" | "rate_limited" | "unavailable" | "timeout" | "invalid_request"` (implement `ErrorCategory::as_str()` in `ai-providers` and use it for both usage rows and API responses).

20. **Cursor pagination for `GET /tenant/ai/usage`** mirrors `conversations::routes`/`queries` exactly: cursor encodes `(created_at, id)` (copy the `encode_cursor`/`decode_cursor` helper pattern from `crates/modules/conversations/src/queries.rs`), page query orders `created_at DESC, id DESC`, fetches `limit + 1` rows to compute `has_more`, default limit 25, max 100.
