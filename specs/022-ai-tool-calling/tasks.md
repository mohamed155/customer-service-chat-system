## How to read a task

Each task names the exact file, the exact function/type signature to add or change, the doc section that fully specifies behavior, and an **Accept:** line stating the check that proves it done. Do not invent behavior not in the referenced section. After finishing a task, change its `- [X]` to `- [X]`.

## Phase 3: User Story 1 — AI answers using an approved tool (P1) 🎯 MVP

**Goal**: The engine offers the tenant's enabled auto-approved built-in tools to the model, executes validated calls (bounded chain, bounded timeout), feeds results back, and records + surfaces every request in the conversation's timeline. Approval-required calls are recorded and pause the chain (full customer-facing resolution is US2).

**Independent Test**: quickstart.md Scenario 1 — enable `lookup_customer`, ask a question requiring it, verify the AI's reply reflects the result, a `succeeded` timeline entry with duration appears and persists after reload, and disabling the tool first produces a `refused` entry with the AI still answering (Scenario 3 step 1).

## Dependencies & Execution Order

- **Setup (T001–T002)** → **Foundational (T003–T012)** must finish before any user story.
- **US1 (T013–T028)** depends only on Foundational. **This is the MVP.**
- **US2 (T029–T046)** depends on US1 (chain/executor/timeline/SSE variant exist).
- **US3 (T047–T051)** depends on US1 (timeline) and US2 (decision fields to surface). Independent of US4.
- **US4 (T052–T065)** depends on US1 (`executor`/`policy` exist) only. Independent of US2/US3 — the `crypto.rs` relocation (T052) touches shared `ai` code but is additive (a re-export shim), so it does not block or get blocked by US2/US3 work in flight.
- **Polish (T066–T068)** last.

Story independence: US3 and US4 can each be built directly on top of US1 (+ US2 for US3's decider field) in either order. US2 is the only story that also touches the outbox/responder claim loop.

## Parallel Opportunities

- Foundational: **T005, T006, T007** (the three vendor mapping files) in parallel; **T008** in parallel with those (different crate); T009→T010→T011→T012 mostly sequential (each builds on the prior file's types) but T009 can run parallel to T004–T008.
- US1: **T020, T021** in parallel (different frontend files); implementation T013→T014→T015 sequential (same crate boundary); T025–T028 tests in parallel once T013–T019 land.
- US2: **T037, T038** in parallel; **T041–T046** tests in parallel once T029–T036 land.
- US3: **T050, T051** in parallel.
- US4: **T061–T065** tests in parallel once T052–T057 land.
- Polish: **T066, T067** in parallel; T068 last.

## MVP Scope

**US1 (T001–T028)** = smallest shippable increment: the AI can call an enabled, safe tool and the result — success or refusal — is fully recorded and visible. Everything after (human approval, richer audit/failure UX, tenant-defined tools and self-service admin) is additive on top of it.

## Implementation Strategy

Build in priority order US1 → US2 → US3 → US4, shipping/validating each story's checkpoint before starting the next. Because every story after US1 is additive and independently testable, work can stop after any checkpoint with a coherent, releasable system — in particular, US1 alone is safe to ship (no side-effecting tool can run without US2, since no approval-required tool can be enabled with any practical effect until then).
